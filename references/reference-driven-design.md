# 레퍼런스 기반 디자인 — 반영 · 철학 수립 · 검증

레퍼런스 이미지(예: "이런 느낌으로", PPT 샘플, 대시보드 스크린샷)를 받아 슬라이드 덱을 만들 때의 표준 절차. **지식샘터 모드·범용 슬라이드 제작 모두에 적용**한다. 2026-07-07 연수① 키컷 제작에서 실측·검증한 교훈을 규정으로 굳힌 것이다.

핵심 흐름은 셋이다: **① 레퍼런스를 시각 언어로 분해 → ② 덱 철학 + 컴포넌트 미니철학 수립 → ③ 정직하게 렌더-검증.** 이 순서를 지키고, 키컷 1장이 검증을 통과하기 전에는 전체 빌드로 넘어가지 않는다.

---

## 1. 레퍼런스 반영 — 시각 언어로 분해(tokenize)

레퍼런스를 "따라 그리는" 것이 아니라 **디자인 언어를 토큰으로 추출**한다. 받은 이미지를 확대해(필요하면 crop) 다음을 명시적으로 읽어낸다:

- **배경**: 플랫인가 그라디언트/대기감인가. 코너 글로우·오로라 여부. (거의 항상 그라디언트다 — 플랫로 오해하지 말 것.)
- **표면(카드/패널)**: 불투명 흰색인가 글래스(반투명+blur)인가. 그림자의 깊이·번짐. 모서리 반경.
- **타이포**: 굵기(얇은 지오메트릭 vs 굵고 둥근), 한글/라틴 분리, 자간, 위계.
- **팔레트**: 주색·보조색·데이터 그라디언트. 몇 색까지 쓰는지(대개 절제).
- **데이터 시각화의 형태 언어**(가장 자주 놓침): 막대라면 — 폭(넓음/얇음), 캡(상단만 둥근/pill/각짐), 색이 막대마다 바뀌는가(가로 색진행), 막대 안 세로 그라디언트, 값 배지 유무, 축·그리드 유무, 발광/그림자. 도넛·꺾은선도 동일하게 형태를 읽는다.
- **요소 종류 구분**: 어디가 (a)레이아웃/텍스트/카드, (b)차트/데이터시각화, (c)일러스트/사진 에셋인지 나눈다. 셋은 제작 경로가 다르다(§3).

추출 결과는 **`design.md`(토큰: 색·폰트·간격·반경) + `PHILOSOPHY.md`(톤 서술)**로 남긴다. 폰트는 `brand-pptx`의 font_discovery로 톤 매칭하고 Pretendard/Noto는 회피한다(AI 느낌). 얇은 지오메트릭이 기본이되 너무 얇으면(Air만) 허전하니 본문은 Regular로 존재감을 준다.

## 2. 디자인 철학 — 덱 철학 + 컴포넌트 미니철학(2층)

철학은 **2층 구조**로 세운다:

1. **덱 철학** — 덱 전체를 지배하는 하나의 `PHILOSOPHY.md`(예: "Whispered Aurora"). 배경·표면·타이포·팔레트의 대원칙. 미감 4원칙을 반드시 포함한다:
   - ① 배경은 그라디언트/대기감(플랫 금지) ② 깊이=강한 다층 그림자+글래스 ③ 빽빽하게 채움(허전 금지) ④ 얇고 정제된 타이포(단 Air만이면 허전 → 본문 Regular).

   > **★ ③ 밀도가 최우선 (구체 실행 — 연수① 사용자 피드백으로 확정).** "여백 최소화, 텍스트·이미지를 그리드/카드 비율에 맞춰 공허하지 않게." 여백은 요소 *사이*의 호흡이지 큰 빈 영역이 아니다. `PHILOSOPHY.md`가 "여백 60%" 같은 문구를 담더라도 **본문 슬라이드에는 밀도 우선이 우세**하도록 쓴다(표지·간지만 더 비울 수 있음). 실행:
   > - **콘텐츠 그리드는 `flex:1`로 캔버스를 채운다.** 콘텐츠를 `justify-content:center` 블록으로 감싸 위아래 큰 빈 공간을 남기지 말 것(이게 실패 원인이었다).
   > - **카드 내부는 분배해 채운다.** 풀하이트 카드에 짧은 행 몇 개를 위에 몰아두는 "빈 늘어난 카드"가 안티패턴. 행 자체를 `flex:1`(균등 분할, 헤어라인 구분)이나 `space-between`으로 채우거나, 아니면 카드를 콘텐츠 높이로 줄인다. 위-콘텐츠+아래-공백 금지.
   > - **타이포·이미지를 셀 비율에 맞춰 키운다.** 본문 13~15px·스탯 크게. 히어로/일러스트는 **에셋 실제 비율에 슬롯을 맞춰**(예 1:1 에셋엔 `aspect-ratio:1/1` 슬롯) 왜곡·레터박스 없이 셀을 채운다.
   > - **공허 테스트**: 본문 슬라이드에 카드-갭보다 큰 빈 영역(텍스트·도형·이미지·의도된 초점 여백이 아닌)이 있으면 주변 콘텐츠를 끌어와 채우거나 키운다. 본문 콘텐츠가 유효 캔버스의 ~85% 미만이면 너무 성기다.
   > - **채우되 정렬·리듬을 깨지 않는다.** 밀도는 캔버스를 쓰는 것이지 겹치도록 욱여넣는 게 아니다. 엣지 정렬·간격 리듬 규칙은 그대로 유효 — 빽빽하고 *동시에* 정밀하게.
2. **컴포넌트 미니철학** — 덱에서 **시그니처가 되는 데이터시각화**(예: 대시보드 히어로 막대 차트)에는 그 컴포넌트만을 위한 짧은 미니철학 `.md`를 따로 쓴다. 이번 연수①의 `AURORA_ASCENDANT.md`(막대=상승하는 빛의 기둥)가 예시다. 미니철학은 canvas-design 스킬의 절차(철학 .md → 캔버스로 표현)로 만들고, 장인 수준의 크래프트를 반복 강조한다. 예시 파일: `assets/examples/component-philosophy-example.md`.

**언제 미니철학까지 가는가**: 레퍼런스의 데이터시각화가 덱의 인상을 좌우하고, 네이티브 차트로는 그 형태 언어(둥근 캡·가로 색진행·값 배지 등)를 못 낼 때. 보조·소형 차트에는 덱 철학만으로 충분하다.

## 3. 차트/에셋 정책 — 하이브리드(확정)

**손으로 CSS/SVG를 그려 렌더하지 않는다**는 원칙은 유지하되, 차트는 **하이브리드**로 간다:

- **히어로·대형 데이터시각화 = 레퍼런스 재현 크래프트 이미지.** 네이티브 차트로 형태 언어가 재현 안 될 때(둥근 캡·가로 색진행·글래스 값배지 등), 컴포넌트 미니철학에 따라 **HTML/CSS로 정밀 제작 → Playwright로 PNG 렌더 → `<img>`/`addImage`로 배치**한다. 템플릿 `assets/templates/component-barchart.html` + `scripts/render_component_png.js` 사용.
- **소형·보조 차트, 또는 값 편집이 중요한 차트 = 네이티브 엑셀 연동 차트**(`pptxgenjs.addChart`). 더블클릭 데이터 편집 가능. 데이터 성격에 맞춰 형태 다양화(분포=막대, 구성비=도넛/원, 추이=꺾은선, 누적=영역). 네이티브 막대 그라디언트는 XML 후처리(chart1.xml의 `<a:solidFill>`→`<a:gradFill>`).
- **한 슬라이드에서 혼용 가능**: 예 — 히어로 막대는 크래프트 이미지, 옆 도넛은 네이티브(연수① 키컷이 이 조합).
- **트레이드오프 명시**: 크래프트 이미지 차트는 PowerPoint에서 값 편집 불가(값 변경 시 소스 HTML 데이터 수정 후 재렌더). 지식샘터 제출물처럼 수치가 확정된 산출물이면 무방하고 미감이 우선. 편집 여지를 남겨야 하면 네이티브를 택한다 — 이 판단은 사용자에게 확인한다.
- **배경 그라디언트 = 생성 PNG**를 슬라이드 배경으로 주입(`slide.background={path:...}`). html2pptx는 CSS 그라디언트 배경을 드롭한다.
- **일러스트·마스코트·개념도 = kie.ai gpt-image-2** 생성. 손 SVG 금지.

## 4. html2pptx 저작 하드 규칙 (레이아웃은 수동 좌표 금지)

레이아웃·카드·텍스트는 html2pptx(브라우저 측정)로 만들어 좌표 드리프트를 없앤다. 단 아래 규칙을 어기면 텍스트가 통째로 증발하거나 빌드가 막힌다 — **모두 실측으로 확인된 함정**이다:

- **차트 슬롯은 `class="placeholder" id="..."`로, 반드시 비운다.** placeholder는 자기 요소만 스킵하고 **자식은 여전히 검증·변환**된다. 슬롯 안에 프리뷰 막대/SVG를 넣으면 각 요소가 에러(예: "Background images on DIV"). html2pptx가 반환하는 `placeholders[]`(인치 좌표)를 받아 변환 후 `slide.addChart(...)` 또는 `slide.addImage(...)`로 그 자리에 삽입한다.
- **모든 텍스트는 `<p>`/`<h1~6>` 안에.** 최상위 `<span>` 텍스트는 렌더되지 않는다(KPI·범례·타일이 통째로 사라짐). span은 `<p>` 안의 run으로만 쓴다.
- **`<div>`에 그라디언트/이미지 배경 금지**(그라디언트 텍스트·장식 막대는 단색화). **텍스트 요소(`<p>`)에 배경 금지** — 알약 버튼은 `<div>`(배경) + 내부 `<p>`(흰 텍스트) 구조로.
- **SVG stroke-path 아이콘은 렌더 불가** → 텍스트 글리프(예 ▲ `&#9650;`)나 kie.ai 이미지로 대체.
- **글래스 카드 깊이감**: base.css의 2겹/음수 spread 그림자는 html2pptx 파서가 **첫 겹만 읽고 음수 spread를 무시**해 흐릿·평평해진다. 변환용 HTML에서 **단일 겹·양수 그림자**(예 `0 13px 30px rgba(46,52,90,.26)`)로 오버라이드하고, 카드 불투명도를 .90~.94로 올려 배경 블리드를 줄인다(브라우저=진실이므로 프리뷰도 같이 바뀜).

## 5. 검증 — 정직한 렌더 루프(키컷 게이트)

**렌더해서 눈으로 확인하기 전에는 "됐다"고 하지 않는다.** 파이프라인:

1. **빌드**: `node build.js`(html2pptx + placeholder 슬롯에 차트/이미지 삽입) → 필요 시 막대 그라디언트 XML 후처리.
2. **렌더**: LibreOffice headless. `soffice.com --headless --convert-to pdf`. 함정 — ① **설치 직후 PC 재시작 전까지 미작동**(bundled Python 미초기화) ② `"Could not find platform independent libraries"`는 **무해한 경고**(무시, PDF는 정상 생성) ③ **한글·공백 경로 회피**(ASCII 경로로 복사) ④ 전용 `-env:UserInstallation` 프로필, 잔류 `.~lock`·`.tmp`·`soffice.bin` 정리 ⑤ PowerPoint COM은 비대화형 세션에서 E_FAIL — 신뢰 불가, 렌더는 LibreOffice로.
3. **PDF→PNG**: pymupdf(fitz) 150dpi.
4. **정직 비교**: 렌더 PNG를 레퍼런스/의도 목업과 나란히 합성(side-by-side)해 본다. **유사도를 과대평가하지 않는다**(연수①에서 "96%" 과장 후 반복 지적받음). 레이아웃·정렬·폰트 두께·차트 형태·글래스 깊이를 냉정히 보고, 남은 차이가 (a)고칠 수 있는 것인지 (b)네이티브/변환 고유 한계인지 구분해 정직하게 보고한다.
5. **키컷 게이트**: 가장 복잡한 슬라이드 1장(대시보드 등)을 먼저 이 루프로 통과시킨다. 통과 전에는 전체 빌드 금지. 통과 후 5장 배치로 진행하며 배치마다 2단계 검증(브라우저 스샷 + LibreOffice PDF).

**크래시 진단 팁**: 특정 PPTX만 LibreOffice에서 크래시(exit -1073740791)하면, 요소를 격리(배경만/차트만/조합)해 원인을 좁힌다. 수동좌표 빌드가 흔한 원인 — html2pptx 아키텍처로 교체하면 대개 해소된다.
