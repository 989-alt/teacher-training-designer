# 슬라이드 자료 제작 표준

연수 설계(Step 0~6)가 끝난 뒤 실제 발표 자료(슬라이드)를 제작할 때 적용한다. 설계 단계와 마찬가지로, 먼저 계획을 세워 사용자 확인을 받은 뒤에만 실제 제작에 들어간다 — 계획과 제작을 분리한다.

## 완결성 기준 (핵심 원칙)

"연수를 전혀 듣지 않은 초보 수강생이 교안만으로 모든 강의 내용을 완벽히 따라갈 수 있을 정도로 자세하게" 만든다. 이 기준을 만족하려면 슬라이드 1장마다 다음 4가지가 **텍스트로도(이미지 없이도)** 파악 가능해야 한다:

1. 지금 보고 있는 화면이 무엇인지 (예: "PowerShell 창 — 설치 명령어를 막 입력한 직후")
2. 무엇을 왜 하는지 (예: "이 명령어는 Node.js 없이도 Claude Code를 설치합니다")
3. 그대로 복사해 쓸 수 있는 정확한 문자열 (명령어·경로 등은 항상 코드블록)
4. 막혔을 때의 대처 ("안 되면 →" 캡션). 전문용어가 처음 등장하면 1줄 정의를 병기한다.

## 분량 산정

"한 스텝 = 한 장면 = 한 슬라이드" 원칙으로 세분화한다. 참고 삼을 수 있는 실측 사례(단일 차시 수업 1편을 다룬 예시 교안)는 44~61슬라이드였다.

다차시 연수를 설계했다면 **분량 기준을 반드시 사용자에게 확인**한다 — "전체 차시 합산 기준"인지 "차시당 기준"인지에 따라 총량이 몇 배 차이 난다. 확인 후 차시/섹션별 예상 장수를 표로 산정해 제시하고 확인받는다. 표 형식 예:

| 차시 | 섹션 | 예상 장수 | 비고 |
|---|---|---|---|
| 1차시 | (섹션명) | (장수) | (비고) |
| **1차시 소계** | | **(합계)** | |
| **총계** | | **(총합)** | |

## 이미지 소싱 3단계 (우선순위 순)

1. **실제 캡처 가능한 것**: 브라우저 기반 UI(로그인 화면, 웹 아티팩트, 웹 서비스 결과물 등)는 브라우저 자동화 도구(mcp__claude-in-chrome__*)로 지금 바로 캡처한다.
2. **생성형 이미지로 대체**: 데스크톱·터미널처럼 직접 캡처할 수 없지만 재현 가능한 장면, 그리고 개념도·비유 다이어그램은 이 단계로 처리한다. 반드시 **실제 명령어 실행 결과(정확한 텍스트)를 먼저 확보**한 뒤, 그 내용이 그대로 보이는 이미지를 생성 요청한다 — 검증되지 않은 내용을 이미지에 넣지 않는다. `brand-pptx` 스킬의 kie.ai 파이프라인을 활용한다.
3. **사용자 직접 캡처**: 위 두 방법으로 확보할 수 없는 것(개인·학교 환경에 의존적인 실제 결과 화면, Claude가 실행 권한이 없는 데스크톱 GUI, 실제 트러블슈팅 사례 등)은 **목록으로 정리해 사용자에게 명시적으로 전달**한다 — 조용히 생략하지 않는다.

이 3단계 분류와 각 슬라이드가 어디에 해당하는지는 `assets/slide-plan-template.md`의 소싱 매트릭스로 정리해 사용자 확인을 받는다.

## 제작 워크플로우

1. **차시별 슬라이드 스크립트 작성** — 제목·본문 텍스트·필요 이미지 태그(어느 소싱 단계인지 포함)를 담은 상세 대본을 차시당 1개 문서로 작성한다.
2. **이미지 소싱 병행** — 위 3단계를 순서대로 처리한다.
3. **html2pptx로 조립** — `pptx.md` 규칙을 따른다: **python-pptx 절대 사용 금지**, html2pptx 워크플로우, 사용자가 다른 테마를 지정하지 않는 한 white minimal이 기본.
4. **차시별 검수** — "이 슬라이드만 보고 실습이 재현되는가"를 기준으로, 완결성 기준 4가지를 슬라이드마다 다시 점검한다.

## 원칙

- 분량과 이미지 소싱 방법은 반드시 사용자 확인을 받은 뒤 실제 제작에 들어간다.
- 이미지 소싱 우선순위(①실캡처 ②생성형 ③사용자 캡처)를 조용히 건너뛰지 않는다 — 특히 ③으로 넘어가는 항목은 반드시 명시적으로 사용자에게 알린다.
- 스크린샷을 가장한 부정확한 명령어·화면을 만들지 않는다.
